# Governance

Glacier Engine uses lightweight, contribution-led governance while the project
is experimental.

## Roles

- **Contributors** improve code, tests, documents, examples, or issue analysis.
- **Reviewers** provide repeated, reliable review in one or more project areas.
- **Maintainers** merge changes, manage releases, enforce community policies,
  and protect architectural invariants.

Roles are earned through sustained, constructive participation. No employer,
vendor, or contribution size grants special control over technical decisions.

## Decisions

Routine changes are decided in pull-request review. Changes to public formats,
security boundaries, evidence semantics, or governance begin with a GitHub issue
that records:

- the problem and scope;
- alternatives considered;
- compatibility and migration effects;
- acceptance and rollback gates.

Maintainers seek rough consensus. When consensus is not possible, they document
the decision and rationale in the issue. Decisions can be revisited when new
evidence appears.

## Becoming a reviewer or maintainer

A maintainer may nominate a contributor who has demonstrated sound judgment,
respectful collaboration, and follow-through. Existing maintainers confirm the
role publicly. The same process can remove an inactive role without removing
credit for past work.

As the community grows, this document should evolve through the same public
proposal process.
